01 · What this is

Overview

The project fine-tunes FinBERT with LoRA adapters to produce two labels per input text — sentiment (positive / neutral / negative) and market impact (bullish / neutral / bearish) — from a single forward pass through one shared encoder with two independent classification heads.

The intended consumer is predict.py, called directly from a market-research pipeline in place of an LLM sentiment call — deterministic, free, and fast enough to run locally on headlines, earnings-call snippets, 8-K passages, and Fed announcement text.

02 · Why the setup looks like this

Hardware & Constraints

Training runs entirely on a CPU laptop — there is no dedicated NVIDIA GPU available, which rules out QLoRA's 4-bit quantization (a CUDA-only path). The project instead uses plain LoRA against a small, finance-pretrained base model, which keeps the trainable footprint tiny without needing quantization at all.

ConstraintConsequence
No CUDA GPUQLoRA is unavailable; plain LoRA adapters used instead
Base model ≤ 500M paramsProsusAI/finbert (110M) chosen over Llama/Mistral/GPT-2-medium class models
Batch size ≤ 8Larger batches don't speed up CPU training; kept fixed at 8
Weights & data off-diskAll artifacts round-trip through Google Drive via drive_storage.py

FinBERT already carries financial vocabulary — earnings terminology, banking-sector language, regulatory phrasing — from its Reuters/Bloomberg/Financial Times pretraining. That means the LoRA adapters only need to learn the six-class label scheme, not financial language itself, which is what makes a small dataset and CPU-only training viable at all.

03 · Model shape

Architecture

A single BFSISentimentModel wraps the FinBERT encoder in a LoRA adapter, pools the [CLS] token, and feeds it into two separate linear heads — one per task. Both predictions come from one encoder pass.

Why query/value only

LoRA adapters attach to the attention query and value projections only, skipping key. Adding key typically improves accuracy by under 1% while adding meaningful CPU training time — not a good trade at this scale.

05 · Training data

Data Pipeline

Training data comes from Financial PhraseBank (sentences_allagree — sentences where all annotators agreed on the label), sourced from HuggingFace. Impact labels aren't in the original dataset, so they're derived programmatically from sentiment.

Getting the data out of HuggingFace

takala/financial_phrasebank still ships as a loading script, and the installed datasets release (5.0.0) has dropped script-based dataset loading entirely — including on HuggingFace's own datasets-server, so no auto-converted Parquet mirror was available either. The fix: download FinancialPhraseBank-v1.0.zip directly from the repo via huggingface_hub.hf_hub_download and parse Sentences_AllAgree.txt by hand (latin-1 encoded, sentence@label per line) — the same data the script would have produced, without the script.

The resulting CSV passed dataset_prep.py's validate_csv cleanly and was uploaded to Drive as training_data.csv.

10 · Incident log

Issues Found & Fixed

Financial PhraseBank wouldn't load

Cause: datasets 5.0.0 dropped support for loading-script datasets; takala/financial_phrasebank still ships one, and no auto-converted Parquet mirror existed.

Fix: downloaded the raw zip via huggingface_hub.hf_hub_download and parsed Sentences_AllAgree.txt directly.

Training crashed on batch 0 — ValueError: too many dimensions 'str'

Cause: in dataset_prep.py, a batched .map() both removed and re-added columns named sentiment_label / impact_label. datasets 5.0.0 unified the output schema back to the original string dtype, silently turning the int-encoded labels back into strings.

Fix: encode labels to ints on the pandas DataFrame before constructing the HF Dataset, avoiding the same-name column collision entirely.

Training process killed twice, no traceback

Cause: attaching a live tail -f monitor to the log file the background training process was writing to caused the process itself to be killed at the same point each time (right after model load).

Fix: ran training in the background without a concurrent tail; it completed cleanly end to end.

Adapter filename mismatch — .bin vs .safetensors

Cause: the installed peft saves adapters as adapter_model.safetensors; drive_storage.py's download_model_from_drive() hardcoded adapter_model.bin, which doesn't exist on Drive.

Fix: try the .safetensors filename first, fall back to .bin. Verified by downloading into a scratch directory and loading the model from it.

Evaluation report showed metrics under the wrong class name

Cause: sklearn's classification_report/confusion_matrix order classes alphabetically unless labels= is passed explicitly — but target_names is applied positionally regardless. evaluate.py passed target_names without labels, so e.g. the row printed as "positive" actually held negative's numbers, and the impact report was rotated by a full position.

Fix: pass an explicit labels= order matching target_names to both calls, and label the confusion matrix axes with that same order.
